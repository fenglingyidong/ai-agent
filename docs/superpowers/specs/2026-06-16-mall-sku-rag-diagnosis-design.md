# Mall SKU RAG 评测诊断设计

## 背景

2026-06-16 的 30 题电商导购 RAG 评测从 `120/300` 提升到 `164/300`，但复杂链路和推荐题仍不稳定。当前不直接修改提示词或代码，先定位失败来自路由、上下文、工具调用、RAG 召回、生成使用，还是记忆污染。

## 目标

对低分且具有代表性的题目做证据化诊断，输出每题根因标签和后续优化优先级。

抽样题目：

| 题号 | 问题类型 | 诊断重点 |
|---|---|---|
| Q08 | 静音鼠标推荐 | 是否调用 RAG，是否命中 `3004` |
| Q09 | 护眼台灯推荐 | 为什么只追问，不推荐 `3015` |
| Q13 | 42 码跑鞋推荐 | 是否被历史偏好污染 |
| Q20 | 机械键盘对比 | 为什么有知识库商品却泛化轴体 |
| Q25 | 桌面组合 | 为什么不做 SKU 组合 |
| Q29 | 厨房组合 | 为什么不做厨房电器加收纳组合 |

## 诊断维度

每题检查五层证据：

1. 路由层：`task_type` 是否合理，是否影响后续链路选择。
2. 上下文层：ReAct 或 SimpleTask 实际收到的 prompt 是否包含干扰信息、历史偏好或错误约束。
3. 工具层：`searchProductKnowledge` 是否注册、是否被调用、调用 query 是否贴近用户需求。
4. RAG 层：召回的 parent/child 内容是否包含期望 SKU，分数和排序是否合理。
5. 生成层：如果召回正确，模型是否正确使用；如果召回错误，失败点回溯到 query 或召回。

## 根因标签

| 标签 | 含义 |
|---|---|
| `NO_TOOL_CALL` | 应查 RAG，但模型未调用 `searchProductKnowledge` |
| `BAD_QUERY` | 调用了 RAG，但 query 过泛、过窄或缺少关键商品词 |
| `BAD_RECALL` | query 合理，但 RAG 未召回目标商品 |
| `GOOD_RECALL_BAD_USE` | 召回正确，但模型没有使用或错误使用 |
| `MEMORY_POLLUTION` | 历史偏好覆盖或干扰本轮明确需求 |
| `WRONG_TOOL_PATH` | 误走商城 MCP 或其他工具导致失败 |
| `PROMPT_GAP` | prompt 规则表达不足，导致追问、泛化或编造 |

## 输出

新增一份诊断报告：

`docs/evaluation/2026-06-16-mall-sku-rag-diagnosis.md`

报告包含：

1. 每题的链路路径、工具调用、RAG 召回和最终回答摘要。
2. 每题根因标签。
3. 共性问题排序。
4. 后续优化建议，分为短期提分、根因治理和观测补强。

## 非目标

本阶段不修改代码、不改提示词、不调整配置、不重跑全量评测。诊断结论经确认后，再进入实现计划。

## 验证

诊断报告必须满足：

1. 每个根因标签都有证据引用。
2. 每个建议能对应至少一个失败题。
3. 不把模型回答本身当作唯一证据，必须结合 trace、工具调用或 RAG 结果。
4. 如果缺少 trace 证据，明确标注为“证据不足”，并给出需要补采的字段。
